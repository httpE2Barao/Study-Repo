🏗 Engenharia de Requisitos

A Engenharia de Requisitos é a base do desenvolvimento de software. Neste estágio, são definidas, documentadas e analisadas as necessidades do sistema, garantindo uma construção eficiente e alinhada às expectativas dos usuários e stakeholders.

🔍 Por que isso é importante?

✅ Redução de riscos e desperdícios 🚀

✅ Comunicação clara entre desenvolvedores e clientes 🗣️

✅ Maior controle sobre mudanças no projeto 📑

📌 2. Visão Geral

A engenharia de requisitos é composta por etapas fundamentais:

1️⃣ Estudo de Viabilidade – Avaliação técnica e econômica do projeto.

2️⃣ Análise de Requisitos – Entendimento profundo do que o sistema deve realizar.

3️⃣ Definição e Especificação de Requisitos – Tradução das necessidades em linguagem técnica.

4️⃣ Documentação de Requisitos – Registro oficial para guiar o desenvolvimento.

💡 Benefício-chave: Essa fase evita ambiguidades e retrabalho, garantindo um desenvolvimento fluido e eficiente.

🎯 3. Tipos de Requisitos

Os requisitos podem ser classificados de diferentes formas:

📌 1. Requisitos de Sistema

🔹 Relacionados à infraestrutura, como hardware e banco de dados.

📌 2. Requisitos de Usuário

🔹 Descrevem o que o usuário espera, de maneira acessível.

📌 3. Requisitos Funcionais

🔹 Especificam o que o sistema deve fazer (exemplo: registrar usuários).

📌 4. Requisitos Não Funcionais

🔹 Definem aspectos como desempenho, segurança e usabilidade.

🕵️‍♂️ 4. Elicitação de Requisitos

✨ 4.1 O que é Elicitação?

A elicitação envolve descobrir, entender e definir requisitos de forma estratégica.

🚧 4.2 Desafios na Elicitação

🔸 Visões Conflitantes – Stakeholders podem discordar sobre funcionalidades.

🔸 Requisitos Implícitos – Nem tudo é declarado claramente.

🔸 Escopo Mal Definido – Pode levar a mudanças constantes.

🔸 Volatilidade – Mudanças no decorrer do projeto são comuns.

💡 Dica: Requisitos mal definidos são um dos principais motivos de falhas no desenvolvimento de software.

🔬 5. Técnicas de Elicitação de Requisitos

🛠️ A escolha da técnica ideal depende do contexto do projeto:

🗣️ 5.1 Entrevistas

Abertas → Stakeholders respondem livremente.

Fechadas → Perguntas estruturadas para respostas objetivas.

📖 5.2 Storytelling

Os usuários contam suas experiências diárias para revelar detalhes importantes.

🎭 5.3 Role-Playing

O analista assume o papel do usuário para entender seus desafios cotidianos.

🔍 5.4 Etnografia

Observação direta do ambiente de trabalho dos usuários.

🤯 5.5 Brainstorming & Mind Mapping

Brainstorming → Geração rápida de ideias 💡

Mind Mapping → Organização visual das informações 📌

🎨 5.6 Prototipação

Criação de modelos iniciais do sistema para validar ideias.

📌 Exemplo Prático: No desenvolvimento de um sistema de logística, um analista pode:

1️⃣ Realizar entrevistas com motoristas 📞

2️⃣ Acompanhar uma rota de entrega 🛻

3️⃣ Organizar um brainstorming com a equipe de TI 🖥️

📖 6. Casos de Uso

📌 História: Introduzidos por Ivair Jacobson em 1987 e incorporados à UML em 1995.

📌 6.1 Componentes dos Casos de Uso

👤 Ator – Quem interage com o sistema.

📝 Caso de Uso – Ações realizadas pelo usuário no sistema.

📌 6.2 Exemplos

📚 Sistema de Biblioteca:

Atores: Cliente, Bibliotecário.

Casos de Uso: Solicitar Empréstimo, Registrar Devolução.

🏨 Sistema de Hotel:

Atores: Cliente, Recepcionista.

Casos de Uso: Fazer Reserva, Consultar Disponibilidade.

🟢 6.3 Benefícios dos Casos de Uso

✅ Definem claramente o escopo do sistema.

✅ Facilitam a implementação e manutenção do código.

✅ Servem como documentação de referência para a equipe.

🎯 7. Conclusão

A Engenharia de Requisitos é fundamental para o sucesso de qualquer projeto de software. Com técnicas eficazes, podemos garantir que o sistema atenda às necessidades reais dos usuários.

📌 Relembrando os principais conceitos:

✅ Definir requisitos de forma clara evita problemas futuros.

✅ Técnicas como entrevistas, storytelling e prototipação melhoram a coleta de informações.

✅ Casos de uso estruturam a modelagem do sistema de forma compreensível.
